    Methodology¶

    For every issue below: - "Fixed" means we have a commit that addresses the reported problem, but the reporter has NOT confirmed it works for them yet. The correct action is to comment on the issue with the fix details and ask for retest — not to close. - "Verified fixed" means the reporter (or another affected user) confirmed resolution. Safe to close. - "Partially addressed" means our work covers some but not all of what the user reported. - "Retest FAILED" means the reporter tested our fix and it did NOT resolve their issue. Needs re-investigation. - Attachments marked MANUAL need you to download from GitHub and review — they are file links I cannot access.

    #178 — Custom endpoint URL ignored (always connects to OpenAI) ⚠️ UPSTREAM BUG¶

    • Reporter: Ywocp | Created: Feb 21 | Status: Hotfix2 partial fix + upstream PySubtrans bug identified
    • Category: Translation/GUI
    • Attachments: OK — full repro steps, detailed retest report with 3 scenarios.
    • Analysis (ORIGINAL): Endpoint passthrough was missing from translate-during-transcription code path.
    • Fix (hotfix1): Commit 8fe643a — Added --translate-endpoint CLI arg, passed to all call sites.
    • Retest Result (2026-02-27, Ywocp): ALL THREE test scenarios FAILED with 404 on /v4/responses.
    • Root cause (VERIFIED in hotfix2): Two layers:
    • (A) URL normalization (our bug, FIXED in hotfix2): Users pasting full endpoint URLs like https://api.example.com/v1/chat/completions caused doubled suffix since OpenAI SDK appends /chat/completions automatically. Fixed: _normalize_api_base() strips trailing API path suffixes in all 3 code paths (core.py, cli.py, service.py).
    • (B) PySubtrans upstream bug (NOT ours to fix): Provider_OpenAI.model_is_reasoning_model() treats every non-GPT model as "reasoning" → routes to OpenAIReasoningClient → client.responses.create() → /responses endpoint. Third-party providers only implement /chat/completions. This causes 404 for ALL non-GPT models (GLM, Qwen, DeepSeek, etc.) on any endpoint.
    • Hotfix2 fix: Commit 8d0deae — URL normalization in core.py, cli.py, service.py. Comment posted to #178 explaining upstream bug.
    • Impact: MEDIUM — URL normalization helps, but upstream PySubtrans bug still blocks non-GPT models on custom endpoints.
    • Action: Upstream bug report to machinewrapped/llm-subtrans pending (user will file). WhisperJAV cannot monkey-patch PySubtrans internals — that's unmaintainable.

    #176 — Translation settings lost after GUI restart ✅ FIXED (hotfix2)¶

    • Reporter: Ywocp | Created: Feb 20 | Status: Fixed in hotfix2, awaiting retest
    • Category: GUI/Persistence
    • Attachments: OK — detailed retest report.
    • Analysis (ORIGINAL): PyWebView private_mode=True causes ephemeral localStorage.
    • Fix (hotfix1): Commit 8fe643a — Set private_mode=False with storage_path. INSUFFICIENT — WebView2 localStorage unreliable between sessions.
    • Root cause (VERIFIED in hotfix2): The JS TranslationSettingsModal only saved to localStorage and never called the backend file persistence API (settings.json). WebView2's localStorage is unreliable across sessions regardless of private_mode setting.
    • Fix (hotfix2): Commit 8d0deae — Dual persistence architecture:
    • api.py: New get_translation_settings() / save_translation_settings() bridge methods with camelCase↔snake_case key mapping. Merges with existing settings to preserve fields the modal doesn't manage.
    • app.js: save() now async — persists to both localStorage (fast sync cache) and backend file (authoritative). New loadSettingsFromBackend() loads from backend file on pywebviewready event (after WebView2 bridge is available).
    • Timing: localStorage loaded sync during DOMContentLoaded, backend file loaded async during pywebviewready (overrides with authoritative data).
    • Impact: MEDIUM — translation settings now persist to ~/.config/WhisperJAV/translate/settings.json.
    • Action: Awaiting retest from reporter.

    Hotfix1 Retest Failure Analysis (resolved in hotfix2)¶

    Two hotfix1 fixes were confirmed NOT working by reporter Ywocp (2026-02-27). Both re-investigated and addressed in hotfix2:

    1. #178 (endpoint passthrough): Hotfix2 added URL normalization to prevent doubled API path suffixes. However, the core 404 issue is an upstream PySubtrans bug — model_is_reasoning_model() misclassifies all non-GPT models as "reasoning", routing them to /responses instead of /chat/completions. This cannot be fixed by WhisperJAV without unmaintainable monkey-patching.

    2. #176 (settings persistence): Hotfix2 bypassed unreliable WebView2 localStorage entirely. New dual persistence: JS saves to both localStorage (sync cache) and backend settings.json (authoritative). Backend loads override localStorage on pywebviewready.
